## 一、项目评估

### 1.1 三个方向的真实评估

**方向 1：精美安装引导 + 好看网站**

坦率说：这是锦上添花，不是雪中送炭。

对标大厂的安装引导（比如 VS Code、Docker Desktop 的安装器），技术上需要用 Electron 或 Tauri 做一个 GUI 安装器，工作量至少 2-4 周。但用户只会经历一次安装，装完就忘了。

**什么时候做**：产品跑通之后，作为"品牌升级"做。但不要指望靠这个改变世界。

---

**方向 2：让模型能力从"玩具"变成"真正能干活"**

这是最核心的方向，但也是最难的。

你现在用的是 Qwen2.5-Coder-7B，8GB 显存。现实是：

| 模型大小 | 能力水平 | 适合场景 |
|----------|----------|----------|
| 7B | 能补全、能简单问答 | 学习、轻量编辑 |
| 14B | 能理解中等复杂度代码 | 日常开发辅助 |
| 32B+ | 能做架构级推理 | 真正的生产力工具 |
| 70B+ | 接近 GPT-4 水平 | 专业级开发 |

7B 模型在 8GB 显存上已经是极限了。要让模型"真正能干活"，有两条路：

- **路 A**：等硬件降价（RTX 6070 16GB 出来后跑 14B，成本会大幅下降）
- **路 B**：做**模型路由**——简单问题用本地 7B，复杂问题调用云端 API（用户可选）

**路 B 才是关键**。你不需要让本地模型干所有事，你需要让用户**感知不到本地和云端的边界**。这才是"改变世界"的方向——不是让 7B 变成 70B，而是让用户永远得到最好的答案，不管它来自哪里。

---

**方向 3：AI 控制电脑的可视化 UI**

这是最有想象力的方向，但也是最拥挤的赛道。

你现在看到的：
- Claude → Computer Use
- OpenAI → Operator
- Cursor → Agent Mode
- Anthropic → Claude Code
- Google → Project Mariner

大厂都在做。但它们有一个共同弱点：**全部依赖云端，全部要联网，全部要付费**。

这就是你的机会。

### 1.2 我看到的真正机会

你的项目有一个大厂做不到的特质：**完全本地、完全免费、完全隐私**。

这就像 1991 年的 Linux——Unix 贵得离谱，Linus 做了一个免费的替代品。现在 Claude/Cursor/GitHub Copilot 每月 $20-$200，你做了一个**不花钱的**。

但"免费"本身不够，Linux 能成功是因为它**真的能干活**。所以你的核心命题是：

> **让本地 AI 真的能干活，让不花钱的用户也能享受 AI 编程的便利。**

### 1.3 项目有潜力吗？有价值吗？

**有。认真说。**

价值不在代码，在场景。让我给你算一笔账：

| 方案 | 月费 | 年费 | 门槛 |
|------|------|------|------|
| GitHub Copilot | $10 | ¥860 | 需要信用卡 |
| Cursor Pro | $20 | ¥1720 | 需要信用卡 |
| Claude Pro | $20 | ¥1720 | 需要信用卡+国际网络 |
| ChatGPT Plus | $20 | ¥1720 | 需要信用卡+国际网络 |
| **FlashTap** | **¥0** | **¥0** | **一台 RTX 笔记本** |

中国有多少大学生有 RTX 笔记本但办不了国际信用卡？**几百万。**

这些人就是你的价值。他们不是"付不起"，是"付不了"。这个鸿沟，大厂不会帮你填——它们赚的就是付费用户的钱。

**三个趋势正在同时发生：**

- **趋势 1：本地模型越来越强** —— 模型能力每年都在跳。你搭的是管道，不是水箱。水箱会换，管道一直在。
- **趋势 2：显存越来越便宜** —— RTX 5060 已经有 8GB，下一代 16GB 可能成为标配。你的项目会自动变强，不需要你做任何事。
- **趋势 3：AI 编程从"奢侈品"变成"必需品"** —— 绝大多数中国学生还在用记事本级别的开发环境。这个差距，FlashTap 可以填。

**但潜力 ≠ 成功。** 三个残酷的事实：

1. **安装体验是生死线** —— 一个用户安装失败，他不会来第二次
2. **7B 模型确实是玩具** —— 第一印象决定留存
3. **一个人做不了所有事** —— 先跑通，再开源，再吸引贡献者

**所以：别想潜力，别想价值，别想改变世界。现在就一件事——找一台空白电脑，跑通它。跑通了，价值自然来。**

## 二、优化路线

### 2.1 初期路线图

```
Phase 1：验证期（现在 → 跑通空白电脑）
├── 修完已知 Bug
├── 找 3-5 个朋友测试
└── 目标：安装成功率 > 90%

Phase 2：能力期（跑通后 1-2 个月）
├── 实现"模型路由"架构
│   ├── 本地 7B：补全、简单问答（免费、隐私）
│   ├── 云端 API：复杂推理、架构设计（用户自备 key）
│   └── 用户无感知切换——同一个对话，自动选最优路径
├── 支持多个本地模型（Qwen / DeepSeek-Coder / CodeLlama）
└── 目标：用户说"这个 AI 真的能帮我写代码"

Phase 3：Agent 期（2-4 个月后）
├── 做一个轻量级 Agent 框架
│   ├── 不做"控制整个电脑"（太危险，大厂也在摸索）
│   ├── 做"控制开发环境"——更安全、更可控
│   │   ├── 读写文件
│   │   ├── 运行命令
│   │   ├── Git 操作
│   │   └── 搜索代码
│   └── 这就是 Cursor Agent Mode 的本地版
├── UI：在 VS Code Continue 插件基础上扩展
│   └── 不需要从零做——Continue 是开源的，可以 fork
└── 目标：用户说"这个 AI 能自己干活了"

Phase 4：生态期（6 个月后）
├── 精美安装器（Tauri，体积小、性能好）
├── 产品网站
├── 插件/扩展市场（让社区贡献 Agent 技能）
└── 目标：用户说"我离不开它了"
```

### 2.2 关于"改变世界"

你说致敬 Linux v0.01，这个类比很好，但要注意 Linux 成功的关键不是 Linus 写了内核，而是：

1. **他解决了真实痛点**（Unix 太贵）
2. **他开放了代码**（社区能贡献）
3. **他坚持了 30 年**（不是一两年就成功的）

FlashTap 如果想走同样的路，核心不是技术多先进，而是：

> **让每一个买得起游戏本的学生，都能拥有一个免费的 AI 编程助手。**

大厂做的是"给专业开发者卖 $20/月的工具"，你做的是"让所有人都能用上 AI 写代码"。这两个是完全不同的市场。

中国有几千万大学生，大部分买不起 Claude Pro，但很多人有 RTX 笔记本。**这才是你的"改变世界"——AI 编程的普惠化。**

**v0.01 这个版本号很好，别急着改。Linus 当年也是从"just a hobby, won't be big and professional"开始的。先跑通，再迭代，坚持做下去，时间会给你答案。**

### 2.3 先跑通再优化

核心原则：**先求稳，再求好。**

现在项目里有一个尚未解决的 Bug（WSL 内 C/C++ 调试器扩展安装失败）。如果现在做优化（拆脚本、去重配置），代码会大量变动，到时候在空白电脑上跑不通，分不清是原有 Bug 还是优化引入的新 Bug——**变量太多，无法定位问题。**

**现在可以做的（低风险，不影响主流程）：**

| 改动 | 风险 | 理由 |
|------|------|------|
| 给 `Setup-FlashTap.ps1` 加函数区域注释分隔 | 零 | 纯注释，不改变执行 |
| 清理 `__pycache__/` 并确保 `.gitignore` 覆盖 | 极低 | 已有规则，只是清理残留 |
| 给 `install-vscode.ps1` 里硬编码的 VS Code 版本号加注释提醒 | 零 | 只是提醒自己，不改逻辑 |

**等跑通再做的（中高风险，会改变运行逻辑）：**

| 改动 | 风险 | 理由 |
|------|------|------|
| 拆分 `Setup-FlashTap.ps1` 为多个子脚本 | 高 | 改变脚本调用链 |
| 配置文件去重（只保留一份源配置） | 中 | Continue 插件对三种格式的读取优先级不确定 |
| 增加卸载脚本 | 中 | 需要在安装成功后才能验证 |
| VS Code 下载链接去版本号硬编码 | 中 | 需要实测镜像是否支持 latest 重定向 |

## 三、本地增强方案

> 模型路由的思路，本质上不还是云端的吗？而云端，不就又回到大厂付费的吗？
> 我要的是：在算力和模型"智商"不变的情况下，最大限度的增强本地模型的代码能力。

### 3.1 核心思路

**7B 模型的"智商"是固定的，但"知识"和"工具"可以无限扩展。**

打个比方：一个初级程序员（7B），如果你让他凭脑子写代码，他只能写简单的。但如果你给他：
- 一整个公司的代码库当参考（RAG）
- Stack Overflow 全部精华回答（知识库）
- 编译器、调试器、搜索引擎当工具（Tool Use）
- 一个资深架构师写好的模板让他套（Prompt Engineering）

他就能干中级甚至高级的活。**他的智商没变，但他的产出变了。**

### 3.2 六种具体方法

#### 方法 1：RAG + 向量数据库 —— 最有效，必须做

这是让本地模型"开卷考试"的核心：

```
用户提问："怎么用 Express 写一个 JWT 认证中间件？"
    ↓
向量搜索本地代码库 + 知识库 → 找到 3 段相关代码
    ↓
把找到的代码塞进 prompt 的上下文
    ↓
7B 模型基于这些上下文生成回答
```

- **没有 RAG**：7B 模型凭训练数据回忆，可能记错 API、用过时的写法
- **有 RAG**：7B 模型看着真实代码照着写，准确率大幅提升

技术选型：

| 组件 | 推荐方案 | 理由 |
|------|----------|------|
| 向量数据库 | ChromaDB | 纯 Python，本地运行，零依赖 |
| Embedding 模型 | nomic-embed-text | 只需 274MB，Ollama 直接跑 |
| 文档切分 | LangChain chunking | 标准方案 |

**关键：Embedding 模型很小（274MB），和 7B 主模型可以共存于 8GB 显存，不需要同时加载。**

#### 方法 2：本地知识库 —— RAG 的"弹药"

RAG 是枪，知识库是子弹。你需要预装高质量的知识源：

| 知识源 | 大小 | 价值 |
|--------|------|------|
| MDN Web Docs 精华摘录 | ~50MB | 前端开发必备 |
| Python 官方文档精要 | ~30MB | Python 开发必备 |
| 常见框架 Best Practices | ~20MB | React/Vue/Express 等 |
| LeetCode 题解精选 | ~10MB | 算法学习 |
| 项目自身的代码库 | 动态 | 最相关的上下文 |

这些都可以离线打包，随 FlashTap 一起安装。用户装完就有"开卷考试"的资料。

#### 方法 3：Tool Use（工具调用）—— 让模型"动手"

这是最被低估的能力。7B 模型可能写不出完美的代码，但如果它能：

```
用户："帮我修复这个 bug"
    ↓
模型调用工具：读取文件 → 分析代码 → 运行测试 → 看报错 → 修改代码 → 再测试
    ↓
通过多轮工具调用，逐步逼近正确答案
```

Ollama 原生支持 Tool Use（function calling），你不需要自己实现。关键是定义好工具集：

| 工具 | 作用 | 实现难度 |
|------|------|----------|
| `read_file` | 读取代码文件 | 简单 |
| `write_file` | 写入代码文件 | 简单 |
| `run_command` | 执行终端命令 | 中等（需要沙箱） |
| `search_code` | 搜索代码库 | 简单（ripgrep） |
| `git_diff` | 查看 Git 变更 | 简单 |
| `run_test` | 运行测试并获取结果 | 中等 |

**效果**：一个 7B 模型 + 6 个工具，比一个 70B 模型"裸跑"更实用。因为 70B 模型也只能一次生成，而 7B + 工具可以**多轮迭代**。

#### 方法 4：精准 System Prompt —— 零成本提升

这是最简单但最被忽视的方法。一个好的 System Prompt 可以让 7B 模型的输出质量提升一个档次：

```
差的 Prompt：
"你是一个编程助手"

好的 Prompt：
"你是一个专注于 {用户的技术栈} 的编程助手。
当用户提问时：
1. 先分析问题的核心需求
2. 给出简洁的实现方案
3. 标注关键代码的用途
4. 如果不确定，明确说明而非猜测
技术栈：{从项目文件自动检测}"
```

**关键**：System Prompt 应该根据用户的项目**动态生成**——检测 `package.json`、`requirements.txt`、`Cargo.toml` 等，自动识别技术栈，告诉模型"你是一个 React + TypeScript 专家"。

#### 方法 5：Speculative Decoding —— 让推理速度翻倍

这不是提升"智商"，而是提升"体验"。7B 模型生成慢，用户等得烦。

原理：用一个小模型（比如 0.5B）快速生成候选 token，大模型并行验证。如果小模型猜对了（通常 60-70% 的概率），就直接采纳，速度提升 2-3 倍。

Ollama 已经在实验这个功能，后续版本可能原生支持。

#### 方法 6：模型微调（Fine-tuning）—— 长期方向

用开源的代码指令数据集（如 CodeFeedback、OpenOrca）对 Qwen2.5-Coder-7B 做 LoRA 微调，让模型更擅长：

- 中文编程问答
- 常见框架的代码生成
- 错误诊断和修复

LoRA 微调只需要 16GB 显存（8GB 也能跑 QLoRA），训练一次约 2-4 小时。微调后的模型可以和原模型合并，不增加推理开销。

**但这不是 v0.01 该做的事，是 v1.0 的方向。**

### 3.3 综合效果预估

| 方案 | 智商提升 | 知识提升 | 体验提升 | 实现难度 |
|------|----------|----------|----------|----------|
| RAG + 知识库 | ⭐ | ⭐⭐⭐⭐⭐ | ⭐⭐ | 中 |
| Tool Use | ⭐⭐⭐ | ⭐⭐ | ⭐⭐⭐⭐ | 中 |
| 动态 System Prompt | ⭐⭐ | ⭐⭐⭐ | ⭐⭐ | 低 |
| Speculative Decoding | - | - | ⭐⭐⭐⭐⭐ | 高（等 Ollama） |
| 微调 | ⭐⭐⭐⭐ | ⭐⭐⭐ | - | 高 |

### 3.4 实施顺序

```
Phase 1（现在，和跑通同步）：
└── 动态 System Prompt
    └── 检测项目技术栈 → 自动生成专业化的 Prompt
    └── 成本：几乎为零
    └── 效果：立竿见影

Phase 2（跑通后 1 个月内）：
└── RAG + 本地知识库
    ├── 安装 nomic-embed-text（274MB）
    ├── 安装 ChromaDB
    ├── 预装编程知识库（~100MB）
    └── 自动索引用户项目代码

Phase 3（2-3 个月后）：
└── Tool Use
    ├── 定义工具集（read/write/run/search）
    ├── 修改 Continue 配置启用工具调用
    └── 实现安全的命令执行沙箱

Phase 4（长期）：
└── 微调专用模型
└── Speculative Decoding（等 Ollama 原生支持）
```

**最关键的一句话：7B 模型 + RAG + Tool Use > 70B 模型裸跑。**

这不是瞎说，这是已经被多篇论文验证的结论。RAG 让模型有知识，Tool Use 让模型有手，System Prompt 让模型有方向。一个有知识、有手、有方向的初级程序员，比一个只会背书的博士更实用。

**关于"算力不变增强 AI"是不是科研方向**——是的，而且是非常前沿的方向：

| 方向 | 关键词 | 代表工作 |
|------|--------|----------|
| 检索增强生成 | RAG, Retrieval-Augmented Generation | Lewis et al. 2020 |
| 工具增强 LLM | Tool-augmented LLM, Tool Learning | Toolformer (Schick et al. 2023) |
| 小模型增强 | Small Model Enhancement, Model Routing | FrugalGPT (Chen et al. 2023) |
| 上下文学习 | In-context Learning | GPT-3 Paper (Brown et al. 2020) |
| Agent 系统 | LLM Agent, Autonomous Agent | AutoGPT, LangChain |

### 4.1 每一步后的真实水平

### 6.2 一个人全栈开发的三个原则

**原则 1：你不需要懂每一行代码，但必须懂架构**

```
你不需要知道：PowerShell 的 $LASTEXITCODE 怎么工作
你必须知道：  安装流程有哪几步，每一步的输入输出是什么
```

具体做法：每次让 AI 写代码之前，先自己画一个流程图（哪怕在纸上画）：

```
安装流程：
  bat 提权 → ps1 主脚本 → install-ollama → install-vscode → download-model → configure → check
  每一步：输入什么 → 做什么 → 输出什么 → 失败了怎么办
```

这个流程图就是你的"架构理解"。AI 帮你填代码，你负责确保架构正确。

**原则 2：每次只让 AI 做一件事**

最常见的新手错误：一次给 AI 一个大需求，比如"帮我实现 RAG 功能"。AI 会给你生成 500 行代码，你看不懂，也测不了，出 bug 不知道在哪。

正确做法：把大需求拆成最小的可验证单元：

```
❌ "帮我实现 RAG"
✅ "帮我写一个 Python 函数，输入一段文本，输出文本的 chunk 列表，每个 chunk 不超过 500 字"
   → 你能测试：随便给一段文字，看输出是不是 chunk 列表
   → 你能验证：chunk 长度是不是 <= 500

✅ "帮我写一个函数，调用 Ollama 的 nomic-embed-text 模型，输入一段文本，输出向量"
   → 你能测试：运行一下，看返回的是不是数字列表
   → 你能验证：向量维度是不是 768
```

**每一步都独立可测试。一个函数测通了，再写下一个。**

**原则 3：测试不是"跑一遍看行不行"，是"写断言"**

```python
# ❌ 手动测试：运行脚本，看输出，人眼判断
# ✅ 自动测试：写断言，跑一下，全绿就过

def test_chunk_text():
    result = chunk_text("Hello world " * 100, max_length=500)
    assert len(result) > 1, "应该被分成多个 chunk"
    assert all(len(c) <= 500 for c in result), "每个 chunk 不超过 500 字"
    print("✅ chunk_text 测试通过")

def test_get_embedding():
    result = get_embedding("Hello world")
    assert isinstance(result, list), "应该返回列表"
    assert len(result) == 768, "nomic-embed-text 输出 768 维"
    assert all(isinstance(x, float) for x in result), "每个元素是 float"
    print("✅ get_embedding 测试通过")
```

你不需要学 pytest（虽然以后可以学），最简单的 `assert` 就够了。让 AI 帮你写测试函数，你只需要理解"这个函数期望什么输入、什么输出"，然后跑一下。

### 6.3 具体工作流

**写新功能的流程：**

```
第 1 步：你画架构（5 分钟）
    在纸上或文本文件里写：
    "我要做 RAG。需要 3 个函数：
     1. chunk_text(text) → chunks
     2. get_embedding(text) → vector
     3. search_similar(query, db) → results
     然后在 Continue 里配置 embeddingsProvider"

第 2 步：逐个让 AI 实现（每个 10-15 分钟）
    你："帮我写 chunk_text 函数，输入文本，输出不超过 500 字的 chunk 列表"
    AI：生成代码
    你：复制到文件，运行测试

第 3 步：写测试验证（每个 5 分钟）
    你："帮我写一个测试函数，验证 chunk_text 的输出格式和长度"
    AI：生成 assert 代码
    你：运行，全绿就过

第 4 步：组装（10 分钟）
    把测通的函数拼起来，跑端到端流程
```

**调试 bug 的流程：**

```
第 1 步：定位问题范围
    不是"整个脚本不工作"，而是"第几步失败了"
    看日志文件（你已经写了日志系统，这是你做对的最重要的事之一）

第 2 步：缩小到最小复现
    把出问题的代码单独拎出来，不是在 1162 行脚本里调

第 3 步：让 AI 读报错信息
    直接把报错贴给 AI："这个报错是什么意思？怎么修？"
    比你自己猜快 10 倍

第 4 步：验证修复
    改完之后，只跑出问题的那一步，不要从头跑整个安装流程
```

### 6.4 关于"我不会写代码"

说句可能让你意外的：**你比你以为的更会写代码。**

你写了 1162 行 PowerShell 安装脚本，处理了 UAC 提权、空白账户隔离、多镜像下载、PE 文件头校验。这些不是"不会写代码"的人能做到的。

你缺的不是编程能力，是**计算机基础知识**：

| 你现在缺的 | 为什么重要 | 怎么补 |
|------------|-----------|--------|
| 操作系统概念 | 理解进程、权限、环境变量 | 大二会学，现在不急 |
| 网络基础 | 理解 HTTP、代理、DNS | DEVLOG 已经在补了 |
| 数据结构 | 理解向量、哈希、树 | 大二会学，RAG 用到向量时再深入 |
| Python 基础 | 你已经会了，只是不熟练 | 做项目的过程中自然提升 |

**最好的补课方式就是做项目。** 每踩一个坑，就学一个知识点。DEVLOG 就是你最好的教材。

**一条虽然不固定但很实用的路：**

> **架构你来定，代码 AI 来写，测试断言来保。**

- 你定架构 → 你理解全局
- AI 写代码 → 你不需要手写每一行
- 断言保底 → 你不需要看懂每一行，只需要知道"测通了就行"

这三件事，你现在已经会做第一件（你的项目架构很清晰）和第三件（你的日志和自检脚本就是测试）。你只需要把第二件（AI 写代码）从"一次写一大坨"改成"一次写一个小函数"。

**这就是你的路。不是"学完再干"，是"边干边学，每个小步都验证"。**

## 七、不懂架构怎么办

### 7.1 核心方法：去"偷"

"偷"不是抄袭，是**找参考**。所有好的架构都不是凭空发明的，都是改出来的。

**第 1 步：找一个"已经跑起来的东西"**

你想做 RAG？不要从零设计。先找一个已经实现了 RAG 的开源项目，看它怎么做的。

| 你想做的 | 去看什么 | 怎么找 |
|----------|----------|--------|
| RAG | Continue 插件自己的 RAG 实现 | GitHub: continuedev/continue |
| Tool Use | Ollama 官方的 tool calling 示例 | Ollama 官方文档 |
| 向量数据库 | ChromaDB 官方 tutorial | docs.trychroma.com |
| 模型路由 | FrugalGPT 论文 | 搜 "FrugalGPT" |

**关键：不要看教程，看代码。** 教程会教你概念，但不会告诉你"数据从哪来、到哪去、中间怎么处理"。代码会。

### 7.2 画"数据流图"而不是"架构图"

你不需要理解"RAG 架构"这种抽象概念。你只需要回答一个问题：

**数据从哪来？经过什么处理？到哪去？**

以 RAG 为例：

```
用户提问："怎么用 Express 写登录接口"
    │
    ↓
[第 1 步] 把问题变成数字（Embedding）
    输入：一段文字
    处理：调用 nomic-embed-text 模型
    输出：768 个数字（向量）
    │
    ↓
[第 2 步] 用这 768 个数字去找相似的内容（向量搜索）
    输入：768 个数字
    处理：在 ChromaDB 里搜索距离最近的文档
    输出：3 段最相关的文档片段
    │
    ↓
[第 3 步] 把找到的文档塞给模型（上下文注入）
    输入：用户问题 + 3 段文档
    处理：拼成一个完整的 prompt
    输出：模型基于真实文档生成回答
```

**这就是 RAG 的全部架构。没有更复杂的东西了。** 你不需要理解"为什么向量搜索能找到相似内容"（那是数学问题），你只需要知道"输入什么、输出什么"。

### 7.3 让 AI 帮你填代码

画完数据流图之后，逐个让 AI 实现：

```
你："帮我写一个 Python 函数，调用 Ollama 的 nomic-embed-text 模型，
    输入一段文字，输出 768 维向量列表。用 requests 库调用 Ollama API。"
AI：生成代码
你：运行，看输出是不是 768 个数字 → 测通了

你："帮我写一个函数，用 ChromaDB 存入文档向量，并能根据查询向量搜索最相似的 3 个文档"
AI：生成代码
你：运行，看输出是不是 3 段文本 → 测通了

你："帮我写一个函数，把用户问题和搜索到的文档拼成一个完整的 prompt，
    发给 Ollama 的 qwen2.5-coder 模型，返回回答"
AI：生成代码
你：运行，看模型是不是基于文档回答了 → 测通了

→ 三个函数拼起来，RAG 就跑起来了
```

### 7.4 通用 Prompt 模板

当你面对一个完全陌生的领域，直接问 AI：

> "我想在 FlashTap 项目里实现 [RAG/Tool Use/xxx]。
> 请帮我画出数据流图：数据从哪来、经过什么处理、到哪去。
> 不要给我代码，先给我流程。每一步只说：输入是什么、做什么、输出是什么。"

这个 prompt 的关键是**不让 AI 直接给代码**。先要流程，确认你理解了，再逐步要代码。

**用你的 FlashTap 经验来理解**——其实你已经做过一次了：

- Ollama 怎么安装？→ 查资料，发现可以静默安装
- Continue 怎么配置？→ 查文档，发现要三种格式
- 模型怎么下载？→ 查 ModelScope，发现可以用 snapshot_download
- 环境变量怎么设？→ 踩坑，发现 UAC 提权后用户上下文丢了

你不是先学会再做的，你是边做边学的。RAG 也一样。

**一句话总结：不懂架构的时候，不要自己想，去查别人怎么做的，画数据流图，然后让 AI 逐个填代码。架构不是发明出来的，是抄出来再改出来的。** Linus 写 Linux 也不是凭空发明的，他先读了 Minix 的源码，理解了操作系统的数据流，然后自己重写了一遍。你的 FlashTap 也是——你先理解了"装 Ollama → 装 VS Code → 下模型 → 配插件"这个流程，然后写代码实现。RAG 只是另一个流程，方法完全一样。
